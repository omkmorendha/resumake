1. Overview & goals

Resumake is a personal tool that lets the owner upload a LaTeX (or PDF) resume, receive structured, section-anchored feedback from an LLM (optionally tailored to a pasted job description), hold a focused conversation on each feedback point, and let an agent loop propose edits to the underlying .tex — which the user reviews as a diff before applying. The resume compiles to PDF server-side and exports as either .tex or .pdf.

Primary goals (v1)

  • Upload a .tex resume → compile → live PDF preview.
  • Generate structured feedback points (category + severity + section anchor + issue + suggestion), optionally tailored to a job description via structured extraction.
  • Per-point conversation thread; the agent can propose .tex edits applied via diff-approve.
  • Export .tex and compiled .pdf.
  • PDF import → text/structure extraction → pour into one clean built-in LaTeX template.

Non-goals (v1)

  • Multi-user / hosted SaaS / shared accounts (explicitly out — see constraint in §15).
  • Visual fidelity match on PDF import (text+structure into a clean template only).
  • Job-board URL scraping.
  • Heavy compile sandboxing (trusted, self-authored local input).

5. Data model & on-disk layout

Files-only persistence. Each resume is a project directory. JSON sidecars hold feedback, conversations, and version history. Versions are full .tex snapshots (resumes are small).

Durability: all mutating writes (resume.tex, feedback.json, conversation files) use write-to-temp-then-rename for atomicity. On an applied edit the version snapshot is persisted before resume.tex is overwritten, so a crash mid-write always leaves a recoverable prior version. On project load, the app scans versions/ and rebuilds index.json if missing/out-of-sync, warning on orphaned files. summary in the version index is an auto-generated one-line description of the applied diff(s).
Single-tab guard: the app assumes one browser tab per project (no write locking). On open, a project acquires a short-lived session token in project.json; a second tab on the same project shows a "already open elsewhere" warning rather than risking concurrent-write corruption.
data/
└── projects/
    └── <projectId>/                 # e.g. ulid
        ├── project.json             # metadata (name, provider, createdAt, currentVersion)
        ├── resume.tex               # the working LaTeX source (current)
        ├── resume.pdf               # last successful compile
        ├── compile.log              # last compile stdout/stderr
        ├── jobposting.json          # { rawText, extractedRequirements } | absent
        ├── feedback.json            # FeedbackPoint[] (latest analysis)
        ├── versions/
        │   ├── 0001.tex             # snapshot before each applied edit
        │   ├── 0002.tex
        │   └── index.json           # [{ version, ts, summary, byPoint? }]
        └── conversations/

Types (TypeScript / Zod-validated)

type Severity = "critical" | "high" | "medium" | "low" | "nit";
type Category = "impact" | "clarity" | "ats" | "formatting"
              | "relevance" | "consistency" | "grammar";

interface SectionAnchor {
  sectionId: string;        // stable id from the parser (e.g. "experience")
  sectionTitle: string;     // "Experience"
  subBlock?: string;        // "Job 1 · Acme Corp" | bullet index
  texRange?: { start: number; end: number };  // best-effort char range (informational)
}

interface FeedbackPoint {
  id: string;
  category: Category;
  severity: Severity;
  anchor: SectionAnchor;
  issue: string;            // what's wrong / weak
  suggestion: string;       // concrete fix (may include rewritten bullet)
  jdRelevance?: string;     // why it matters for the pasted JD (if any)
  status: "open" | "addressed" | "dismissed";
  // Auto-set to "addressed" when an approved edit targets this point's sectionId
  // (user can manually revert to "open"). "dismissed" is always manual.
}

interface ChatMessage {
  id: string; role: "user" | "assistant" | "tool";
  content: string; ts: string;
  proposedEdit?: { diff: string; targetSectionId?: string }; // surfaced for approval
}

interface JobRequirements {
  mustHaveSkills: string[]; niceToHaveSkills: string[];
  yearsExperience?: string; keywords: string[];
  responsibilities: string[]; rawText: string;
}

6. AI provider abstraction & auth

Interface

interface LLMProvider {
  // Structured, non-agentic call (feedback + JD extraction)
  generateStructured<T>(args: {
    system: string; user: string; schema: ZodSchema<T>;
  }): Promise<T>;

  // Agentic conversation loop with constrained tools; streams events
  runAgentTurn(args: {
    system: string;
    messages: ChatMessage[];
    tools: ResumeTool[];          // app-defined (see §9)
    onEvent: (e: AgentEvent) => void;  // streamed tokens, tool calls, proposed edits
  }): Promise<ChatMessage>;
}

// Tool + event contracts (referenced by runAgentTurn)
type ResumeToolName = "read_resume" | "get_context" | "propose_edit" | "recompile";
interface ResumeTool {
  name: ResumeToolName;
  description: string;
  inputSchema: ZodSchema<unknown>;            // validated before execution
  execute: (input: unknown) => Promise<ToolResult>;
}
type ToolResult =
  | { ok: true; data: unknown }
  | { ok: false; error: string };             // e.g. compile error fed back to the loop

type AgentEvent =
  | { type: "token"; text: string }                                 // streamed assistant text
  | { type: "tool_call"; tool: ResumeToolName; input: unknown }
  | { type: "tool_result"; tool: ResumeToolName; result: ToolResult }
  | { type: "proposed_edit"; diff: string; targetSectionId?: string; rationale: string }
  | { type: "error"; message: string }
  | { type: "done"; message: ChatMessage };
  • ClaudeProvider — wraps @anthropic-ai/claude-agent-sdk's query(). The four ResumeTools are registered as in-process SDK MCP tools; built-in Read/Edit/Bash are excluded from allowedTools. SDK message stream is mapped to AgentEvents. Auth via CLAUDE_CODE_OAUTH_TOKEN. For generateStructured, run a single non-tool turn and Zod-validate (retry policy below).
  • OpenAIProvider — wraps the openai SDK with a manual tool loop. The same four ResumeTools are exposed as OpenAI function tools (their inputSchema → JSON schema). Loop: model emits tool_call → app validates input with Zod → execute() → push tool_result (including {ok:false,error} on failure so the model can react) → repeat until the model returns final text. response_format/JSON schema is used only for generateStructured, not the agentic loop. Auth via pasted OPENAI_API_KEY stored server-side in data/config.json (mode 0600, never sent to the browser, never logged).
  • Zod retry policy (both providers, generateStructured): up to 3 attempts; on each failure, re-prompt with the validation error appended. After the 3rd failure, surface a user-facing "structured extraction failed — try rewording" message and log the raw output.
Auth footgun (documented in official caveats): if ANTHROPIC_API_KEY is set in the environment, the Agent SDK uses it (billed as API credits) instead of your subscription OAuth token. The app must (a) generate the token via claude setup-token, (b) read it from data/config.json or .env as CLAUDE_CODE_OAUTH_TOKEN, and (c) explicitly unset ANTHROPIC_API_KEY in the spawned SDK environment. Add a startup check that warns if ANTHROPIC_API_KEY is present.
Hard licensing constraint: Anthropic does not permit third-party developers to offer claude.ai/subscription login to other end users without prior approval. This tool is and must remain a personal, single-user, local app authenticated with the owner's own token. Do not evolve it into a hosted multi-user product sharing this auth path.

7. Feedback generation & system prompts

Two LLM operations: (a) optional JD requirement extraction, (b) resume analysis producing FeedbackPoint[]. Both use generateStructured with Zod schemas.

Standard resume-review system prompt (synthesized from researched best practices)

Drawn from common high-quality patterns: expert-reviewer framing, ATS-awareness, STAR structure, action-verb + quantification checks, one-column ATS-safe formatting, and per-section Good/Improve/Add analysis. Stored in lib/prompts/resumeReview.ts.

You are an expert technical resume reviewer and career coach who also
understands how Applicant Tracking Systems (ATS) parse resumes. You give
specific, actionable, honest feedback — never vague praise.

Evaluate the resume across these dimensions:
1. IMPACT     — Does each bullet show a quantified result? Strong action
                verb first? STAR-shaped (Situation/Task/Action/Result)?
2. CLARITY    — Concise, jargon-appropriate, no filler ("responsible for").
3. ATS        — One-column, no tables/text-boxes/graphics that break parsers;
                standard section headings; parseable dates.
4. RELEVANCE  — {if JD provided} alignment to the target role's must-haves
                and keywords; {else} alignment to the apparent target role.
5. FORMATTING — Consistency (tense, punctuation, date format), length
                (prefer one page), whitespace.
6. CONSISTENCY/GRAMMAR — tense agreement, typos, parallel structure.

For EACH issue, emit a structured feedback point with:
- category, severity, the section it targets,
- a precise description of the issue,
- a concrete suggestion (rewrite weak bullets verbatim where useful),
- {if JD} why it matters for the target role.

Prefer fewer, higher-signal points over exhaustive nitpicks. Order by
severity. Quote the exact resume text you are critiquing so it can be
anchored to the source. Do NOT invent achievements or fabricate metrics —
flag missing metrics and ask the user to supply them.

JD extraction prompt (when a job posting is pasted)

Extract structured requirements from this job posting: must-have skills,
nice-to-have skills, years of experience, the top 10–15 ATS keywords,
and key responsibilities. Output strict JSON. Do not infer requirements
not present in the text.

The extracted JobRequirements is injected into the review prompt's RELEVANCE dimension, and the review additionally emits a gap analysis (which must-haves/keywords are missing from the resume) as high-severity relevance feedback points.

Conversation / agent system prompt (per-point chat)

You are helping the user improve ONE specific part of their resume:
"{feedbackPoint.issue}" in section "{anchor.sectionTitle}".
You can read the current LaTeX, and propose edits via the propose_edit
tool. Always show the user a focused change; never rewrite unrelated
sections. Keep the resume ATS-safe and one page. When you propose an
edit, briefly explain why. {if JD: optimize toward the target role.}

8. Section anchoring (heuristic LaTeX parsing)

A lightweight parser scans resume.tex and produces a section tree without a full LaTeX parse:

  • Detect \section{...}, \subsection{...}, common resume macros (\resumeSubheading, \cventry, etc.) and environments (itemize, \begin{rSection}).
  • Each detected heading → a SectionAnchor with a stable sectionId (slug of title) and char range to the next heading.
  • Bullets (\item) within a section become candidate sub-blocks.
  • The parser is best-effort: texRange is informational (for editor highlighting); anchoring of feedback uses the stable sectionId + quoted text, which survives edits even when char offsets shift.
  • Fallback when no headings found (custom/exotic document class): the whole body becomes a single synthetic section (sectionId: "document"); feedback anchors then rely solely on quoted-text matching. The app never errors on an unparseable class — it degrades to whole-document anchoring.
  • Multi-level nesting (\subsubsection) is flattened to the nearest section/subsection for anchoring; deeper levels are recorded as subBlock labels only.
  • Anchor staleness: after multiple edits to one section, quoted text may no longer match. The UI hints "results may have shifted — re-run analysis to refresh"; a full re-analysis is the supported way to resync.
Because the agent edits via propose_edit (string find/replace within a section), char ranges drift after edits. Re-run the parser after every applied edit and re-resolve anchors by matching quoted text → section. This is why anchors store quoted text, not just offsets (resolves the brittleness of pure line/char ranges).

9. Agent loop & constrained tools

The agent (Claude via Agent SDK, or OpenAI via manual loop) is given a fixed set of app-defined tools — no Bash, no arbitrary filesystem. For ClaudeProvider these are registered as in-process SDK MCP tools; built-in tools (Read/Edit/Bash) are not in allowedTools.

ToolInputEffect
read_resume—Returns current resume.tex + parsed section tree.
get_context{ sectionId? }Returns: the current FeedbackPoint, full JobRequirements (if present), the target section text + a few adjacent lines, and the prior messages in this point's conversation thread (for coherence). Bounded to keep token use low.
propose_edit{ sectionId, find, replace, rationale }Produces a diff surfaced to the user for approval. Does NOT write to disk.
recompile—Compiles the proposed (or current) .tex to validate it builds; returns errors if any.

Approval gate: propose_edit only stages a diff. The UI shows it; on user approve, the server snapshots the current .tex to versions/NNNN.tex, applies the change, recompiles, updates resume.pdf, and re-parses anchors. On reject, nothing is written. The feedback point this edit targets auto-advances to addressed. On reject, nothing is written. This realizes the "diff preview + approve" decision and keeps the loop observable.

Self-healing compile loop (on post-edit compile failure)

If the recompile after an applied edit fails, the server does not immediately give up:

  1. The LaTeX error (first error + line, from compile.log) is fed back to the agent as a recompile {ok:false,error} result.
  2. The agent attempts a fix via propose_edit → auto-applied within this repair loop → recompile. Up to 3 repair attempts.
  3. If it compiles within 3 tries: success, PDF updates, the repair diffs are folded into the version history with a note.
  4. If it still fails after 3 tries: stop, surface the LaTeX error prominently, and offer one-click undo to the pre-edit snapshot. The edited .tex is left in place (not auto-reverted) so the user can also fix it by hand.
The repair loop's propose_edit calls are auto-applied (no per-hunk gate) only while resolving a compile failure the user already approved the originating edit for — this keeps the fix flow fast without surprising the user with unrelated changes. Each repair attempt is logged.
Streaming: agent events (assistant tokens, tool calls, proposed diffs) stream to the browser over SSE so the chat feels live. Tool execution happens server-side; only propose_edit requires a human gate before it has any effect.

10. LaTeX compilation & PDF

  • Compile via latexmk -pdf -interaction=nonstopmode -halt-on-error in a temp working dir, spawned with child_process.
  • Timeout (e.g. 30s) and process kill on overrun — the one safety measure (input is trusted/self-authored per decision).
  • On success: copy PDF to resume.pdf, log to compile.log, return success + (optional) page count.
  • On failure: parse latexmk/.log for the first error + line number; surface in the UI near the source pane. The PDF pane shows the error message; if a prior good resume.pdf exists it's shown behind a "stale — last compiled successfully <time>" banner, else the pane shows the compile error only (never silently blank).
  • The full TeX Live install lives in the Docker image so arbitrary resume templates/packages compile.
  • Docker preflight: on app startup the compile service checks the Docker daemon (e.g. docker info). If Docker is missing or not running, the UI shows a friendly setup guide (link to Docker Desktop + the one-line image pull) instead of a cryptic runtime error. Docker is documented as a hard prerequisite in the README.
  • Timeout is 30s default, adjustable in config.json (post-v1 UI). On overrun the process is killed and reported as a compile failure.
Compilation is the slowest op (1–5s typical). Debounce auto-compile on source edits, or compile on explicit "Recompile" + after each applied agent edit. Decision: compile on explicit action + after applied edits (not on every keystroke) to keep it snappy. See OQ-2.

11. PDF import (lossy reconstruction)

Honest, achievable path: extract text + infer structure → pour into ONE clean built-in LaTeX template. Output is well-formatted but does not visually match the original PDF.

  1. Extract text with pdftotext -layout (poppler) and/or pdf.js.
  2. LLM generateStructured call segments text into a ResumeSegmentation (Zod-validated):
    { contact: {name, email, phone?, links[]}, summary?: string, experience: {company, role, dates, bullets[]}[], education: {school, degree, dates}[], skills: string[], projects?: {name, desc, bullets[]}[] }.
  3. Render that structure through the bundled "Jake's Resume" one-column template (in lib/templates/; its custom macros are vendored with attribution + license) → resume.tex.
  4. Compile and present. The user can then edit/iterate normally.
The UI must set expectations explicitly: "Imported from PDF — formatting is rebuilt from a clean template, not copied from your original." This prevents the disappointment risk of PDF import.

14. Key flows

Analyze (JD-aware)

UI → POST /jd (paste) → extract JobRequirements → UI → POST /analyze {provider, jd} → generateStructured(reviewPrompt + requirements) → validate FeedbackPoint[] (Zod) → persist feedback.json → render list (sorted by severity, gap items high).

Discuss a point & apply an edit

Select point → POST /chat/:pointId (SSE) → agent runAgentTurn(tools) → [read_resume / get_context] → streams reply → propose_edit(diff) → UI diff modal → user approves → POST /edits/apply → snapshot vNNNN → write resume.tex → recompile → re-parse anchors → PDF + feedback refresh → point status optionally → "addressed".

PDF import

POST /projects (PDF) → pdftotext → generateStructured(segmentation) → render clean template → resume.tex → compile → open project with "rebuilt from template" banner.
